# 公司领域知识地图

## 第一部分：公司业务领域总述

本知识库服务于项目定制研发、AI 编程、需求分析、产品设计和架构设计场景。

用户通常会输入一段客户语言描述，而不是标准产品术语。例如：

```text
我要做一个访客预约审批，扫码入园，离园自动核销功能
```

领域知识服务系统会在当前 Git 版本下读取本文件，判断需求涉及的业务领域、行业线、子系统和知识文档，再读取对应 Markdown 文档，组装成 Agent 可用的 `Knowledge Context`。

核心原则：

```text
Git 管知识版本
knowledge-map.md 管 AI 路由
Markdown 管领域知识
后端服务管读取、路由和组装
知识回流机制负责持续修正
```

## 第四部分：AI 路由规则

1. 需求包含访客、来访、预约、审批、二维码、离园核销时，优先匹配 `企业业务/智慧园区/访客子系统`。
2. 需求包含扫码入园、门禁、闸机、放行、通行权限时，关联匹配 `企业业务/智慧园区/门禁子系统`。
3. 需求包含人员、组织、部门、楼栋、区域、字典、基础档案时，优先匹配 `企业业务/智慧园区/基础管理子系统`。
4. 需求包含车辆、车牌、停车、道闸、月租、访客车辆时，优先匹配 `企业业务/智慧园区/车辆管理子系统`。
5. 需求包含水电气热、抄表、计量、能耗分析、异常用能时，优先匹配 `企业业务/智慧园区/能耗管理子系统`。
6. 需求包含视频监控、摄像头、安防告警、事件联动时，优先匹配 `企业业务/智慧园区/安防监控子系统`。
7. 一个需求可能涉及多个子系统，必须识别一个主子系统，可以识别 0 到 3 个关联子系统。
8. 不要只根据关键词匹配，应优先结合“业务能力描述”和“客户常见表达”做语义判断。
9. 如果无法确定领域，应返回置信度较低的候选领域，并在 reason 中说明需要人工确认。
